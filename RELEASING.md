# Releasing symtether

For maintainers. This describes the mechanics of cutting a release
and what happens after.

## Summary

The version bump is the release trigger.

1. Open a PR titled `chore: release vX.Y.Z` that bumps
   `package.json.version` and `package-lock.json` to the new version.
2. Review, merge to `main`.
3. Everything else is automatic: tag creation, GitHub Release with
   auto-generated notes, npm publish with provenance via OIDC.

There is no manual `git tag`, no `npm version`, no `npm publish`.

## Versioning

Follow [semver](https://semver.org):

- **PATCH** (`0.2.0` → `0.2.1`) — bug fixes only, no behavior
  change users need to know about.
- **MINOR** (`0.2.0` → `0.3.0`) — new features or additive
  behavior, no breaking changes.
- **MAJOR** (`0.2.0` → `1.0.0`) — breaking changes to the CLI,
  the library, or the `#sym:` spec.

Any breaking change goes in a MAJOR release. Breaking changes
include, but are not limited to: CLI flag or subcommand renames,
removed or reshaped exit codes, changes to the `--json` output
schema (`schemas/check-output.schema.json`), incompatible changes
to the `symtether.sum` line format, incompatible changes to the
`#sym:` spec in SPEC.md, and removed or renamed library exports.

Additive changes are MINOR: new commands or flags with defaults
that preserve prior behavior, new tier-1 grammar support, new
fields in `--json` output (consumers must ignore unknown fields),
and new spec sections that do not invalidate previously-valid
`#sym:` references.

## Cutting a release

From a clean `main`:

```sh
git switch main && git pull
git switch -c chore/release-vX.Y.Z
npm version --no-git-tag-version X.Y.Z
git add package.json package-lock.json
git commit -m "chore: release vX.Y.Z"
git push -u origin chore/release-vX.Y.Z
gh pr create --title "chore: release vX.Y.Z" --body "..." --base main
```

The PR should be tiny: two files changed, both with a version bump
on a single line.

Once merged, the CI takes over.

## What CI does

Merging the release PR triggers three things in order:

**1. `release-on-version-bump.yml`** runs on push to `main`. It
compares `package.json.version` on `HEAD` and `HEAD^`. When the
values differ, it calls `gh release create vX.Y.Z --generate-notes
--target <sha>`. GitHub creates the tag from the merge commit and
publishes a release with autogenerated notes (What's Changed +
New Contributors, grouped by PR).

**2. The `release: published` event** triggers `publish.yml`. That
workflow:

- Waits for maintainer approval on the `npm-publish` environment.
  (This is a manual gate — you get a GitHub notification, and you
  have to click "Approve and deploy" in the Actions tab.)
- Runs `npm ci`, build, lint, tests, `publint`, `attw`.
- Verifies `package.json.version` matches the release tag.
- Runs `npm publish --provenance --access public`. The npm registry
  receives the package with a provenance attestation from GitHub's
  OIDC provider.

**3. npm updates.** Within a minute of publish success,
`https://www.npmjs.com/package/symtether` shows the new version and
the ✓ provenance badge.

## Verifying the release

```sh
npm view symtether                    # shows the new version
npm view symtether time.<version>     # timestamp of publish
```

## When things go wrong

**Publish job stuck on "Waiting for approval":** by design.
`npm-publish` has a required-reviewer rule as an extra safety net.
Approve from the Actions tab. If you want to remove the manual
gate, edit the environment in repo Settings.

**`publint` or `attw` fails:** the current build has a
package-config regression. Fix on `main` and re-cut a new patch
version.

**`Verify package.json version matches release tag` fails:** rare.
It means the tag was cut out of band without a bump PR. Delete the
tag (`git push origin :refs/tags/vX.Y.Z`), delete the GitHub Release,
open a proper bump PR.

**`npm publish` fails with a 403 about trusted publishing:** the
npm-side trusted publisher config does not match the workflow's
identity. Check that `jutaz/symtether`, `publish.yml`, and
`npm-publish` are entered exactly on the npm package's Access page.
The workflow filename must match; if you moved the publish job to a
different file (e.g. splitting from ci.yml), update npm accordingly.

## Yanking a release

If a release goes out and turns out to be broken:

- **Within 72 hours:** `npm unpublish symtether@X.Y.Z`. Use this
  window sparingly. Once unpublished, the version number cannot be
  reused.
- **After 72 hours:** you cannot unpublish. Publish a patch release
  (`X.Y.Z+1`) that reverts or fixes the bad code, and mark the bad
  version as deprecated: `npm deprecate symtether@X.Y.Z "reason"`.

## Never

- **Never publish from a laptop.** OIDC via CI is the only supported
  path. The npm account has 2FA on "authorization and writes", which
  blocks `npm publish` from a local session anyway.
- **Never tag manually with `git tag vX.Y.Z && git push --tags`.**
  The version-bump PR is the trigger. Manual tags out of order will
  fail the version-tag check.
- **Never edit a published version's tarball.** Once a version is
  on npm, it is immutable. Publish a new version.
